# ADR-003: MVP実行計画（売れるまでの作業計画）

## ステータス

承認

## コンテキスト

- 初期ターゲットは ADR-002 で BtoC（個人事業主向け）3業種に決定済み。
- ただし、実行順序が未固定で「次に何をすべきか」が曖昧な状態。
- 初期コストを抑えながら、最短で「公開完了率」と「有料化率」を検証する必要がある。

## 決定

MVPは以下の順序で実行する。

1. **対象業種を1つに絞る**（初週）
2. **MVP最小機能を縦に通す**（チャット生成 → 編集 → 公開）
3. **KPI計測を同時実装**（公開率・公開時間・有料化率）
4. **5ユーザー検証を実施**して改善項目を確定

### 実行原則

- 機能追加より先に、公開完了までの体験を1本通す
- 新機能は KPI 仮説が説明できるものだけ採用
- 初期は低規制カテゴリのみ（ADR-002の3業種）

## 作業計画（4週間）

### Week 1: 検証準備とMVP境界確定

- 対象業種を1つに確定（推奨: オンライン講座・コーチング）
- テンプレート3本の初期版を確定（体験申込 / 相談予約 / 資料DL）
- 成功指標の閾値を確定
  - `time-to-first-publish <= 30分`
  - `draft-to-publish rate >= 40%`

### Week 2: 縦切りMVPの実装

- チャット入力からLP初稿生成までを実装
- 編集（見出し・CTA・順序）を実装
- 公開URL発行までを実装

### Week 3: 計測と改善サイクル

- イベント計測を追加（view / click / publish / regenerate）
- ダッシュボードで KPI 可視化
- 5ユーザーで操作ログを取得し、離脱ポイントを特定

### Week 4: 有料化検証

- Starter価格での試験提供（小規模）
- trial-to-paid の検証
- 解約理由・不採用理由を収集し、優先度付き改善バックログを作成

## 選択肢

### 選択肢A: 機能を先に広げる
- **メリット**: 見た目の進捗が早い
- **デメリット**: 公開完了率・有料化率の改善に直結しにくい

### 選択肢B: 1業種に絞って公開体験を先に完成（採用）
- **メリット**: 検証が早く、学習速度が高い
- **デメリット**: 初期ターゲット市場が狭く見える

## 結果

- 開発と検証の優先順位が明確になる
- 売れるかどうかの判定を4週間で実施できる
- 次フェーズの拡張（業種追加）判断をデータで行える

## 判定基準（Go / No-Go）

4週間時点で以下を満たす場合、継続開発（Go）。

- `draft-to-publish rate >= 40%`
- `time-to-first-publish <= 30分`
- `trial-to-paid conversion >= 5%`

いずれかを満たさない場合は、以下を実施する。

- テンプレート訴求の見直し
- ターゲット業種の再選定（ADR-002の次候補へ）
- 価格/プラン構成の再検証
