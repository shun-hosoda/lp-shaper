## 実装方針

### アーキテクチャ

```
UI (page.tsx)
  ↓ form submit
Server Action (lp.ts)
  ↓ Zod validate
  ↓ Supabase server client (auth.uid()で権限確認)
  ↓ DB操作 (projects / page_versions / lp_events)
  ↓ return { data } | { error }
UI (state更新)
```

### コーディング規約

- 戻り値: `{ data: T } | { error: string }` の直和型に統一（`auth.ts` と同じパターン）
- バリデーション: Zodの `safeParse` を必ず使う
- 認証: `createClient()` → `getUser()` → エラー時は `{ error: '認証が必要です' }` を返す
- DB操作: Supabase JS Client を使用（生SQLは書かない）

### テスト戦略

- バリデーション: 正常系3ケース + 異常系3ケース以上
- Server Actions: Supabaseクライアントをモック → ユニットテスト
- UI E2E: playwright は実装後に追加（MVP段階はunitテストのみ）
- カバレッジ目標: Server Actions + Validations = 80%以上

## 議論のポイント（決定事項）

| 論点 | 決定 |
|------|------|
| LLM生成は今回含むか | 含まない。手動JSONでの骨格実装を先に完成させる |
| version_no採番 | `MAX + 1` で実装。レースコンディションはMVP後に対応 |
| publishの原子性 | `projects.status` と `page_versions.state` を逐次更新（Supabase JSトランザクション非サポートのため許容） |
| RLS | 必須。マイグレーションSQLに `auth.uid() = owner_user_id` ポリシーを含める |
| イベント計測 | `project.created` と `published` を今回から記録（draft-to-publish rate計測要件） |
| LP編集UI | テキスト入力のみの最小構成。ビジュアルエディターは後フェーズ |
