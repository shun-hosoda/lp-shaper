# レビューチェックリスト

レビューボードが議論の中で確認すべき項目。
全項目をチェックリストとして順に埋めるのではなく、議論の材料として使う。

## 設計フェーズ（Gate 1）

### PM観点
- [ ] PRDの機能要件がすべてAPI仕様に反映されている
- [ ] ユーザーストーリーが具体的で測定可能
- [ ] MVPスコープが適切（過大でも過小でもない）
- [ ] 変更が「どのペルソナの、どの課題」を解くかが明確
- [ ] 初期ターゲット（B2B SaaS）への適合性がある

### Webマーケ観点（追加）
- [ ] LPの訴求軸（誰に/何を/なぜ今）が一貫している
- [ ] CTA導線が1ページ内で明確（主CTAが分散していない）
- [ ] 計測イベント（表示・クリック・公開）がKPI取得に十分
- [ ] A/Bテスト可能な差分設計になっている

### Architect観点
- [ ] コンポーネント間の依存関係が一方向
- [ ] DBスキーマがAPI仕様のデータモデルと整合
- [ ] 非機能要件が設計に数値目標付きで反映されている
- [ ] ADRで主要な技術選定の根拠が記録されている

## 実装フェーズ（Gate 2）

### Engineer観点
- [ ] 設計書と実装が一致している
- [ ] エラーハンドリングが網羅的（ハッピーパス以外も）
- [ ] テストカバレッジが主要パスを網羅
- [ ] DRY原則に従い重複コードがない

### Security観点
- [ ] 全ユーザー入力がサニタイズされている
- [ ] SQLクエリがパラメータ化されている
- [ ] 認証・認可が適切に実装されている
- [ ] 機密情報がログ・レスポンスに露出していない

### DB観点
- [ ] インデックスがクエリパターンに対応している
- [ ] N+1クエリが発生しない設計
- [ ] トランザクション境界が適切
- [ ] マイグレーションにロールバック手順がある

## リリースフェーズ（Gate 3）

- [ ] 全テストがCI上でパスしている
- [ ] 環境変数・シークレットが適切に管理されている
- [ ] パフォーマンス基準（PRDの非機能要件）を達成
- [ ] ロールバック・障害時の復旧手順が明確

## ビジネス判定（Gate 4: Sellability）

- [ ] `time-to-first-publish` の悪化がない（または改善している）
- [ ] `draft-to-publish rate` 改善の仮説が明記されている
- [ ] `trial-to-paid conversion` への寄与が説明できる
- [ ] `cost-per-publish` が許容範囲内、または改善策がある
- [ ] 失敗時の学習計画（何を見て次に直すか）が定義されている
