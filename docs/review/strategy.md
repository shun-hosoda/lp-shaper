# レビュー戦略

## レビューボード体制

5人の専門家が**合議制**でレビューを行う。各自が独立にチェックリストを埋めるのではなく、
互いの指摘に反応・補足・反論し、議論を通じて品質を高める。

### 専門家の役割

| 専門家 | 主な責務 | 他の専門家との連携ポイント |
|--------|----------|--------------------------|
| Product Manager | 要件整合性・ユーザー価値 | Archと機能スコープ、Engと実現可能性を議論 |
| Architect | 構造・疎結合・拡張性 | DBとデータフロー、Secと認証設計を議論 |
| Senior Engineer | コード品質・保守性・テスト | Archと実装方針、DBとクエリ実装を議論 |
| Security Specialist | 脆弱性・認証認可・データ保護 | Engと実装、DBとデータアクセス制御を議論 |
| DB Specialist | スキーマ設計・クエリ性能・整合性 | Archとデータモデル、Secとアクセス制御を議論 |

## ペルソナの適用

`persona.md` でドメイン固有のペルソナが設定されている場合、
各専門家はそのドメイン知識を持つ人物として議論に参加する。

例: 銀行向けアプリの場合
- PMは「金融庁検査を意識した要件精査」を行う
- Securityは「PCI-DSS準拠」の観点で発言する
- DBは「数億件の取引データ」を前提にクエリを評価する

## マイルストーンゲート

| ゲート | レビュー対象 | APPROVE条件 |
|--------|-------------|-------------|
| Gate 1: 設計完了 | PRD + API仕様 + DB設計 | PM + Arch が APPROVE |
| Gate 2: 実装完了 | コード差分 + テスト結果 | Eng + Sec が APPROVE |
| Gate 3: リリース判定 | CI結果 + 性能テスト | 全員 APPROVE |

## コンテキスト効率化

- 全コードを読むのではなく、**差分（diff）のみ**をレビューする
- PRが存在する場合は**PRコメントの文脈**を踏まえてレビューする
- 500行超の差分は**ファイル単位で分割**してレビューする
- 3ラウンド未解決の指摘は**ADR起票**にエスカレーションする
