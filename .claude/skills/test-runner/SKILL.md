# Test Runner — テスト実行・分析・修正

## 概要

テストランナーを使用してテストを実行し、結果を分析・レビューする。
失敗したテストは自動的に修正を試み、再実行する。

## テスト分類

| レベル | 対象 | 実行時間 |
|---|---|---|
| 単体テスト | 個別モジュール・関数 | 秒単位 |
| 統合テスト | 複数コンポーネント統合 | 数秒〜数十秒 |
| E2Eテスト | 実環境での完全フロー | 分単位 |

## コマンド

### /test (単体テストのみ)
最も高速。個別モジュールのロジックを検証。

### /test --integration (統合テスト)
複数コンポーネントの連携を検証。

### /test --e2e (E2Eテスト)
実環境での完全フローを検証。

### /test --all (全テスト)
全レベルのテストを実行。CI/CD用。

### /test --failed (失敗したテストのみ再実行)
前回失敗したテストのみを再実行。

## テスト実行ワークフロー

### Phase 1: テスト実行
テストランナーでテストを実行し、結果を収集。

### Phase 2: 結果分析
失敗したテストがある場合、エラーパターンを分析。

### Phase 3: 自動修正
失敗理由に基づいて修正を試みる。

### Phase 4: 再テスト
修正後にテストを再実行して確認。

## エラーパターンと修正戦略

| エラータイプ | 検出パターン | 修正戦略 |
|---|---|---|
| AsyncMock設定ミス | `coroutine object` in error | `.return_value` → `AsyncMock(return_value=...)` |
| モック未await | `was never awaited` | `mock.return_value` → `await mock()` |
| バリデーションエラー | `ValidationError` | 制約を緩和、またはテストデータを修正 |
| 環境変数不足 | `ConfigurationError` | `.env` または `monkeypatch` を確認 |
| Import エラー | `ModuleNotFoundError` | 依存関係をインストール |

## 出力フォーマット

### テスト成功時
```
✅ テスト成功
実行: <test_dir> (<N> tests)
結果: N passed, 0 failed
実行時間: X.Xs
```

### テスト失敗時
```
❌ テスト失敗
失敗テスト: N件
各テストの原因と修正方針を出力
```

## テストカバレッジ目標

プロジェクトに応じて設定。一般的な目標は 80% 以上。
